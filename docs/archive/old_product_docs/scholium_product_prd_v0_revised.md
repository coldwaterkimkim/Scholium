# 1. 제품 한 줄 정의

**Scholium은 강의 PDF/슬라이드 위에 자동 생성된 클릭형 해설 앵커를 붙여, 학습자가 자료 밖으로 나가지 않고 현재 문맥 안에서 이해를 이어가게 만드는 학습 인터페이스다.**

------

# 2. 제품이 해결하려는 핵심 문제

대학 강의자료, 특히 STEM∙Health 계열 PDF/슬라이드는 압축적이다.
 학생은 다음 상황에서 자주 막힌다.

- 용어 자체를 모름
- 표/그래프/도식이 왜 중요한지 모름
- 이 페이지가 전체 흐름에서 무슨 역할인지 모름
- 앞 슬라이드 개념이 빠져 있어서 이해가 안 됨
- 교수의 구두 설명이 빠져 있어 맥락이 비어 있음

이때 학생은 보통 자료 밖으로 나간다.

- 구글 검색
- ChatGPT/Perplexity 질문
- 유튜브 검색
- 친구/조교 질문
- 교재/이전 슬라이드 뒤짐

문제는 “검색을 했다”가 아니라, **학습 문맥이 깨진다**는 점이다.
 Scholium은 이걸 더 좋은 검색으로 풀지 않고, **자료 안에 해설을 붙이는 방식**으로 푼다.

------

# 3. 이번 MVP의 목표

이번 MVP의 목표는 크지 않다. 정확히 이거다.

**“PDF 위에 자동 생성된 클릭형 해설 앵커가 실제로 유용한가?”**
 **“학생이 자료 밖으로 안 나가도 될 것 같다고 느끼는가?”**

즉 이번 MVP는 다음을 검증하기 위한 도구다.

1. 페이지 내 핵심 막힘 포인트를 잘 잡는가
2. 그 포인트에 대한 설명이 유용한가
3. 페이지 역할과 앞뒤 문맥 연결이 실제로 도움이 되는가
4. 이 경험이 실제 사용자에게 가치로 느껴지는가

------

# 4. 제품 원칙

## 4.1 핵심 제품 원칙

1. 요약보다 **막힘 해소**가 우선이다
2. 모든 요소를 다 보여주는 것보다 **핵심 포인트를 잘 보여주는 것**이 더 중요하다
3. 내부적으로는 많이 뽑고, 사용자에겐 적게 보여준다
4. 설명은 **짧은 설명 / 긴 설명** 2층 구조로 간다
5. 설명은 “뜻”에서 끝나지 않고, 최소한 **이 페이지에서의 역할**까지 간다
6. hover보다 click이 낫다
7. 지금은 제품 완성도가 아니라 **작동하는 핵심 경험**이 중요하다

## 4.2 구현 원칙

1. **code-first**로 간다
2. n8n은 현재 필수 아님
3. direct PDF LLM input보다 **페이지 이미지 기반 처리**를 우선한다
4. 파이프라인은 **페이지 1차 분석 → 문서 구조 합성 → 페이지 2차 보정** 구조로 간다
5. 분석 결과는 markdown이 아니라 **구조화된 JSON**으로 저장한다


## 4.3 공통 메타데이터 및 처리 표준

### [v0 필수] 공통 메타데이터 규칙

- 모든 JSON 산출물의 `meta`에는 `schema_version`과 `generated_at`를 포함한다.
- AI가 직접 생성한 산출물의 `meta`에는 `prompt_version`과 `model_name`을 포함한다.
- 동일 문서를 재분석할 때, 어떤 프롬프트/모델 조합으로 생성된 결과인지 추적 가능해야 한다.

### [v0 필수] 렌더링 표준

- 분석용 페이지 이미지는 PNG, RGB 기준으로 생성한다.
- 분석에 사용되는 페이지 이미지는 긴 변 기준 표준 해상도로 정규화한 뒤 사용한다.
- 페이지 회전 정보는 렌더 단계에서 정규화하여 분석 입력 이미지는 항상 읽기 쉬운 방향으로 맞춘다.
- bbox는 분석에 사용된 렌더 이미지 기준으로 계산한다.

### [v0 필수] bbox 좌표계 표준

- 좌표 원점은 페이지 이미지 좌상단 `(0, 0)`이다.
- bbox는 normalized `[x, y, w, h]` 형식으로 저장한다.
- 렌더링 시 viewer 크기에 맞춰 픽셀 좌표로 변환한다.
- 기본 박스 기준은 CropBox 우선이며, CropBox가 없거나 비정상일 경우 MediaBox를 fallback으로 사용한다.
- 페이지 회전이 있는 경우 회전을 보정한 뒤 bbox를 계산하고, 원본 페이지 번호는 유지한다.

### [v0 필수] 장식 요소 제외 규칙

- 기본 제외 대상: 학교/기관 로고, 워터마크, 단순 배경 도형, 장식선, 반복 페이지 번호, 단순 브랜딩 아이콘
- 기본 포함 대상: 축 라벨, 범례, 단위, 캡션, 표 헤더, 의미 있는 번호/기호
- 장식처럼 보이더라도 페이지 의미를 바꾸는 경우에는 후보 요소로 포함할 수 있다.

# 6. MVP에서 다루는 요소 체계

우리는 요소를 아래 3층 구조로 본다.

## [v0 필수] A. 텍스트·기호 요소

- 단어
- 문장
- 수식
- 축 라벨
- 범례
- 캡션
- 다이어그램 안 텍스트

## [v0 필수] B. 구조화된 시각 요소

- 표
- 그래프
- 다이어그램
- 화살표 구조
- 프로세스 도식
- 일반 이미지

## [후속 확장] C. 종합 추론 요소

- 텍스트와 그래프의 관계
- 표와 주장 간 관계
- 이 페이지가 앞/뒤 페이지와 맺는 관계
- 생략된 맥락 복원
- 암묵적 의미/함의 해석

중요한 점:

- 내부 품질 평가에서는 **많은 후보 요소**를 뽑아본다
- 사용자에게는 **핵심 3~5개만 노출**한다

# 7. 사용자 경험 정의

## 핵심 경험

사용자는 PDF를 업로드하고, 각 페이지를 보면서 해설 앵커를 클릭한다.
 클릭하면 다음을 본다.

- 이게 무엇인지
- 왜 중요한지
- 이 페이지에서 무슨 역할인지
- 필요하면 더 자세한 설명
- 관련 페이지

즉 UX의 핵심은:
 **“지금 막힌 포인트를 자료 안에서 바로 푼다”**

------

# 8. 사용자 시나리오

## [v0 필수] 시나리오 1: 시험 직전 복습

사용자가 시험 범위 PDF를 올린다.
 페이지마다 3~5개 해설 포인트가 보인다.
 사용자는 그래프/도식/문장/수식 중 막히는 포인트를 눌러 설명을 본다.
 외부 검색 없이 복습을 이어간다.

## [v0 필수] 시나리오 2: 페이지 역할 파악

사용자가 어느 페이지에 와서 “이 페이지 왜 있는 거지?”라고 느낀다.
 페이지 역할 1줄과 핵심 설명을 보고 그 페이지의 위치를 파악한다.

## [v0 필수] 시나리오 3: 앞뒤 페이지 연결 복구

현재 페이지 설명에 관련 페이지가 함께 뜬다.
 사용자는 이전 페이지로 이동해 빠진 맥락을 복구한다.

## [후속 확장] 시나리오 4: 사용자가 직접 텍스트를 선택해서 뜻 확인

모든 텍스트를 미리 앵커로 노출하진 않지만, 특정 텍스트를 선택하면 설명을 받을 수 있다.

## [출시 이후 필요] 시나리오 5: 교수/조교가 수업 전체에 배포

교수/조교가 자기 수업 PDF에 해설 레이어를 붙여 학생들과 공유한다.

------

# 10. 기능 요구사항

## 10.1 업로드 및 렌더링

### [v0 필수] FR-1 PDF 업로드

사용자는 PDF 파일을 업로드할 수 있어야 한다.

### [v0 필수] FR-2 페이지 렌더링

시스템은 PDF를 페이지별 이미지로 렌더링해야 한다.

### [v0 필수] FR-3 작업 상태 관리

시스템은 문서 처리 상태를 저장해야 한다.
 상태 예시:

- uploaded
- rendering
- analyzing
- completed
- failed

### [후속 확장] FR-4 대용량/긴 문서 최적화

긴 문서 업로드 시 처리 효율을 개선한다.

### [출시 이후 필요] FR-5 다중 파일 관리

사용자가 여러 문서를 계정 단위로 관리할 수 있어야 한다.

------

## 10.2 분석 엔진

### [v0 필수] FR-6 페이지 1차 분석

각 페이지에 대해 다음을 생성해야 한다.

- page_role 초안
- page_summary 초안
- candidate anchors 8~15개
- 각 앵커 bbox
- short explanation
- confidence

### [v0 필수] FR-7 문서 구조 합성

페이지 요약들을 바탕으로 다음을 생성해야 한다.

- overall_topic
- overall_summary
- sections
- key concepts
- difficult pages 초안

### [v0 필수] FR-8 페이지 2차 보정

문서 구조를 참고해 다음을 생성해야 한다.

- 최종 노출용 anchors 3~5개
- related pages 1~2개
- long explanation
- prerequisite 보정

### [후속 확장] FR-9 텍스트 선택 기반 설명

사용자가 특정 텍스트를 선택하면 on-demand 설명을 받을 수 있어야 한다.

### [후속 확장] FR-10 복잡한 종합 추론 앵커

한 요소가 아니라 여러 요소의 관계를 설명하는 앵커를 생성할 수 있어야 한다.

### [출시 이후 필요] FR-11 사용자 피드백 반영형 앵커 개선

클릭/체류/만족도 데이터를 활용해 앵커 품질을 개선한다.

------

## 10.3 앵커 및 설명 UX

### [v0 필수] FR-12 앵커 렌더링

각 페이지에 최종 앵커 3~5개를 오버레이로 렌더링해야 한다.

### [v0 필수] FR-13 click-only 인터랙션

앵커는 hover가 아니라 click으로 동작해야 한다.

### [v0 필수] FR-14 2층 설명 구조

각 앵커는 최소 다음을 제공해야 한다.

- 짧은 설명
- 자세한 설명

### [v0 필수] FR-15 뜻 + 역할 설명

설명은 단순 뜻에서 끝나지 않고, 최소한 “이 페이지에서 왜 중요한지”까지 포함해야 한다.

### [후속 확장] FR-16 설명 타입 분류

설명을 다음 타입으로 분리할 수 있어야 한다.

- 어휘형
- 객체형
- 종합형

### [후속 확장] FR-17 추가 확장 정보

일부 개념에 대해 optional expansion 제공

- 더 깊은 맥락
- 관련 개념
- 경우에 따라 어원/배경

### [출시 이후 필요] FR-18 사용자별 설명 깊이 조정

초심자/중급자/숙련자 모드에 따라 설명 밀도를 다르게 제공한다.

------

# 11. 앵커 전략

## 11.1 내부 vs 외부 전략

### [v0 필수] 내부 후보 생성

페이지당 8~15개 candidate anchors 생성

### [v0 필수] 사용자 노출

페이지당 3~5개 final anchors만 노출

이유:

- 내부적으로는 coverage 확인이 필요함
- 사용자에겐 sparse surfacing이 더 낫다

------

## 11.2 앵커 우선순위 선정 기준

### [v0 필수] 우선순위 점수 기준

최종 노출용 앵커는 아래 기준으로 rerank한다.

1. 중요도
2. 혼란 가능성
3. 맥락 의존도
4. 비자명성
5. 후속 영향도

### [v0 필수] 다양성 제약

최종 3~5개는 한 종류로만 채우지 않는다.
 권장 구성:

- 핵심 개념 1개
- 로컬 막힘 포인트 1~2개
- 시각자료 핵심 1개
- 맥락 연결 포인트 1개

### [후속 확장]

- 사용자 클릭 로그 반영 reranking
- 문서 유형별 weighting 조정

------

# 12. 데이터 구조


## [v0 필수] 공통 메타데이터 필드

모든 JSON 산출물은 최소 다음 규칙을 따른다.

- 공통 필수: `meta.schema_version`, `meta.generated_at`
- AI 생성 산출물 필수: `meta.prompt_version`, `meta.model_name`
- 동일 문서의 재분석 결과를 비교할 수 있도록 버전 추적 가능해야 한다.

## [v0 필수] document_summary.json

포함 필드:

- schema_version
- generated_at
- prompt_version
- model_name
- result.document_id
- result.overall_topic
- result.overall_summary
- result.sections
- result.key_concepts
- result.difficult_pages

## [v0 필수] page_analysis_pass1.json

## [v0 필수] bbox 저장 방식

bbox는 정규화 좌표 `[x, y, w, h]`로 저장한다.
렌더링 시 픽셀 좌표로 변환한다.

추가 규칙:
- 좌표 원점은 페이지 이미지 좌상단 `(0, 0)`이다.
- bbox는 분석에 사용된 렌더 이미지 기준으로 계산한다.
- 기본 박스 기준은 CropBox 우선, MediaBox fallback이다.
- 회전된 페이지는 렌더 단계에서 먼저 보정한 뒤 bbox를 계산한다.

## [v0 필수] Step 3. Page Analysis Pass 1

각 페이지 단독 분석

### [v0 필수] Step 3-1. 장식 요소 제외 규칙

## [v0 필수] Step 7. View

# 16. 비기능 요구사항

## [v0 필수]

- 백그라운드 처리 + 완료 후 열람
- 실패/분석중 상태 표시
- 일부 페이지 실패가 있어도 전체 문서를 전부 버리지 않는 방향 고려
- 설명 품질과 bbox 안정성을 우선

## [v0 필수] 실패 처리 정책

- 렌더 단계 전체 실패 시 문서 상태는 `failed`로 처리한다.
- pass1 / pass2에서 일부 페이지만 실패한 경우, 성공한 페이지 결과는 유지하고 실패 페이지는 unavailable 상태로 노출한다.
- document synthesis가 실패하더라도 페이지 단위 결과가 있으면 viewer는 페이지 결과를 우선 제공한다.
- silent failure는 허용하지 않는다. 실패 단계와 원인을 최소한 내부 로그로 남겨야 한다.

## [후속 확장]

- 처리 시간 목표 정의
- 원가 최적화
- 문서 길이별 처리 정책
- 모델 fallback 전략

## [출시 이후 필요]

- SLA
- scale 대응
- 비용 기반 플랜 제어
- 멀티테넌시

중요:
 처리 시간은 지금 hard requirement로 못 박지 않는다.
 지금은 **작동과 품질 우선**이다. 성능은 후속 제약조건으로 관리한다.
